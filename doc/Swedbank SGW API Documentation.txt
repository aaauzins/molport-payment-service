Introduction

Swedbank Gateway is a solution that combines banking services with the company’s business software.

Swedbank Gateway is an asynchronous automated data communications channel between client’s Information System (sometimes referred to as ERP) and Swedbank. Client and the bank exchange data asynchronously in real-time, 24 hours a day, making information available regardless of working hours.

Swedbank Gateway Specification
Technically Swedbank Gateway is RESTful API with requests and responses in XML format. Standard HTTPS (HTTP with SSL) protocol is used for communication.

Client sends data requests to the bank with HTTP POST request and receives responses from the bank with HTTP GET request. After receiving a response, client must send confirmation message with HTTP DELETE request. This means that the whole cycle of one message consists of three parts: sending the request, fetching the response and deleting the requested response.

Definitions and abbreviations
SGW – Swedbank Gateway API.

ERP– Customer Information System (IS) or Enterprise Resource Planning (ERP).

ASICE– term ASiC-E is used through the text to denote both XAdES profile and a container format for digitally signed files with extension .asice.

SK – SK ID Solutions AS (CA in Estonia, http://www.sk.ee).

RC – Registru Centras (CA in Lithuania, http://www.registrucentras.lt/en/ ).

LVRTC - Latvia State Radio and Television Centre (CA in Latvia, https://www.eparaksts.lv/en/)

SGW Operator (in SGW service) – the person who mediates transactions on behalf of the client and to whom extend the conditions of the agreement in order to mediate transactions in the necessary volume.

Operator (in E-invoice service) –the person who intermediates the E-invoices of the client - put under E-invoice service.

Agreement-ID - Swedbank Gateway agreement number provided by the bank and used as parameter X-Agreement-ID that uniquely identifies the agreement in Swedbank EE/LV/LT.

API Key (client_id) - customer identification number in SGW API used in message exchange for customer/Operator identification. This Client ID will be generated during onboarding on Swedbank Developer portal.

X-Request-ID - Message request ID set by the customer, which helps to connect request and response data. ID must be unique.

X-Tracking-ID - Bank side unique message ID in Header.

trackingId - Bank side message ID in DELETE request parameters. The same value as X-Tracking-ID in POST/GET message response.

Date - Date and time of request/response:

All dates and time in parameters & request body should be represented in ISO 8601 date or date time format e.g.: ISO 8601 Date - 2019-05-30 ISO 8601 DateTime - 2019-05-30T00:00:00+00:00

All dates in the HTTP headers are represented as RFC 1123 Full Dates Tue, 21 May 2019 14:23:49 GMT

Transport certificate - is the electronic certificate with which the Client´s or Operator´s information system creates the connection with SGW and confirms the Data Files sent to the Bank.

Signing certificate - qualified electronic signature with OCSP confirmation given by this certificate is used for approving payments in SGW channel.

Signing API - service that enable customer to create, sign payment message on the Bank side with simple REST calls via mTLS connection.

mTLS - Mutual TLS is a type of authentication in which the two parties in a connection authenticate each other using the TLS protocol.

Prerequisites for using Swedbank Gateway
To start using Swedbank Gateway services you need:

Conclude Swedbank Gateway service agreement.
You will be issued unique X-Agreement-ID, used in message exchange for customer identification.
Have Transport certificate for connection - Each communication session is authenticated by client’s Transport certificate.
You need trust Public Root certificate and certificate issuer (DigiCert Global G2 TLS RSA SHA256 2020 CA1) for connecting with psd2.api.swedbank.com, that ensures that client is connecting the right service. The “DigiCert Global G2 TLS RSA SHA256 2020 CA1” certificate is available on the following page: https://www.digicert.com/kb/digicert-root-certificates.htm
For access you need register your company in Swedbank Developer portal: https://developer.swedbank.com/admin/app/home.
After registering you will get API Key(client_id), used in message exchange for customer/Operator identification.
Connection schemes
There are two different connection schemes to use Swedbank Gateway services:

Direct connection
Connection via intermediary (via Operator).
If customer has direct connection, then client has their own Transport certificate for authentication and accesses all services directly only for their own use.

If you act as the technical intermediary- Operator and initiating request on behalf of other company, then for such request you always need to add as header parameter AgreementId (‘X-Agreement-ID’) of this company. If you want to access your own accounts, then you should use your own AgreementId. If not provided Agreement-ID, wrong value or authorization is missing (company hasn’t authorized you as their intermediary in agreement with Swedbank or agreement is not active), then API returns HTTP 403 (Forbidden).

Authentication and security

API communication uses certificate-based authentication (X.509 public key infrastructure (PKI) based standard) over HTTPS (HTTP/1.1 over TLS1.2) protocol.

Following certificates are used for Swedbank Gateway services:

Transport certificate – this certificate is used in order to authenticate communication session from client’s information system/ERP to SGW channel. Transport Certificate file extension and format should be used up to client system implementation. No exact requirements from bank side.

NB! Certificate must have minimum following values:

 	Key Usage: Digital Signature, Key Encipherment.

Since certificates are valid only for a certain period, client must monitor the validity of certificate, get a new certificate when it expires and contact the bank to update the service agreement with the new certificate.

For mission critical systems where interruptions are not allowed, support of two certificates can be planned during the switching period. In case the certificate’s private key has been compromised, client should immediately inform the bank and the certificate will be closed.

SGW supports for electronic signature only ASiCE (Associated Signature Containers – Extended) with timestamp (TS) extension .asice. ASICE (.asice) is signature format, where the long-term evidential value of LT (Long Term time stamp) signatures is ensured with a timestamp based on the RFC 3161 standard. This signature format is preferred in all European Union member states and automatically used in DigiDoc4 (the extension .asice).

For more details about formats please read https://www.id.ee/en/article/bdoc-cdoc-and-asice-digidoc-file-formats-4/.

NB! SGW service does not require any changes for ASiCE container creation besides B4B signature validation, thus follow the instructions provided on https://www.id.ee/en/rubriik/for-developer/.

Connection and messaging

he Swedbank Gateway API is asynchronous messaging. Client message container is like an inbox. Client sends a request to the server and then polls the inbox for a response. If there are any messages in the inbox, by default the oldest message is returned. The whole process of one message consists of three parts: sending the request to the bank, receiving responses from the bank and deleting the response.

Flow sample of API request:
POST [base-url]/account-statements?client_id=<API Key> 		# submit statement request

GET [base-url]/messages?client_id=<API Key>        	     		# get response message with statement info

DELETE [base-url]/messages?trackingId=<trackingId>&client_id=<API Key>  	# confirm message received
Base-URL: https://psd2.api.swedbank.com/partner/v1/sgw/

Connection parameters:
Service (for example account statement) related URL

https://psd2.api.swedbank.com/partner/v1/sgw/account-statement

TLS client authentication is mandatory

Transport certificate must be used for TLS client authentication.
HTTP header

o POST request: X-Request-ID, X-Agreement-ID and Date are mandatory

o NB: if POST request contains xml body then additionally Content-Type is mandatory

o GET response: X-Request-ID and Date are mandatory

o DELETE: X-Request-ID and Date are mandatory + trackingId in parameters

Parameter

API Key (client_id) is mandatory in all requests. Is used for customer identification on API side.

trackingId is mandatory only for DELETE request

POST -sending requests to the Bank

POST [base-url]/{service-url}?client_id=<API Key>

Request sending parameters refer always to the service.

Request sample: POST https://psd2.api.swedbank.com/partner/v1/sgw/account-statements?client_id=<API Key>

After successful posting of the request, it can return a positive response or error.

Request parameters:

Parameter - API Key (client_id) - Customer identification number on API side. Will be generated during onboarding by Developer Portal

Header X-Request-ID – Message request ID set by the company, which helps to connect request and response data. ID must be unique.

Header X-Agreement-ID – Swedbank Gateway agreement number provided by the bank that uniquely identifies the agreement.

o SGW direct agreement or in case Customer acting via Operator, SGW End user agreement ID must be used (as HTTP header X-Agreement-ID).

Header Date - Date and Time of request

o All dates in the HTTP headers are represented as RFC 1123 Full Dates Tue, 21 May 2019 14:23:49 GMT

o Header Content-Type - Content type header must be set when sending in .xml or .asice body, which guarantees proper UTF-8 encoding of the body

o Example: application/xml; charset=utf-8 or application/octet-stream; charset=utf-8 in case of .asice format
Request body – .xml or .asice content

NB: General file size limit is 128KB, but there are some endpoints with a file size limit up to 30MB

GET

By default, always the oldest unread (not deleted) message is returned. The response message is delivered as the response body.

GET [base-url]/messages?client_id=<API Key>

Service url: https://psd2.api.swedbank.com/partner/v1/sgw/messages

Receiving response data from the bank consists of two steps:

Message is received in a response of a GET request.
After receiving a message, a confirmation must be sent out with a DELETE request.
If SGW does not receive a confirmation during one hour after the initial GET request, then the response message will be redelivered to the customer on the next GET request.

NB!
The Swedbank Gateway API is asynchronous messaging.
GET requests must be implemented with a reasonable frequency towards the SGW service (e.g. if all messages have been received, then next GET should be sent after 1 minute).
Even though SGW is intended to be a daily online service, then on a periodical implementation the files are to be downloaded in reasonable time (notifications 48h, other files 14 days) otherwise these will be deleted.
Request
GET request mandatory:

Header parameters:
X-Request-ID – Message request ID set by the customer, which helps to connect request and response data. ID must be unique.
Header Date - Date and time of request
All dates in the HTTP headers are represented as RFC 1123 Full Dates Tue, 21 May 2019 14:23:49 GMT
Request parameter:
API Key (client_id) - Customer identification number on API side. Will be generated during onboarding by Developer Portal
NB! As response to the GET message will be downloaded all messages addressed to the certificate used for request. If same certificate is used in multiple agreements or as operator certificate, the concrete agreement can be identified by AgreementId.

NB!! If there are no more messages to be received (response HTTP ERROR 204), then it is mandatory to delay next GET request for minimum time of 1 minute.

Error handling

To avoid cases when the XML files sent to the bank are not valid, bank strongly recommends validating of XML files on the client side before sending them to the bank.

Validation XSD can be found in SGW tech. page: Estonia, Latvia, Lithuania

Additionally, we recommend not using comments in the beginning of the service XML messages, in front of the header tags for format and service, as it may cause message format and type detection failure.

In case request message maximum file size is exceeded, SGW replies with error “Expected number of transactions in the message is too large”. Please find file size limitation info from "Connection and messaging" chapter or contact the bank in order to find out the current limitation and lower the number of transactions per file to fit the criteria

PS: Communication test doesn’t produce error when service is not activated in client SGW agreement.

There are multiple levels of errors.

Level 1 - PSD2 API level
Level 2 - SGW API level
Level 3 - Service related error - covered under chapter Error codes

Account statement

Customer can use this service to request a list of transactions on account in a given period.

The period range can be up to 90 days. If the period also includes current day, the request will be divided to 2 responses (camt.052.001.02 for current day and camt.053.001.02 for past dates) and will be charged – also the service Current Day Account Statement has to be active in SGW agreement.

Account statement is returned in one or more responses with one response containing up to 10 000 transaction records. This is a current limit which might change. Limit can differ based on the format requested.

Response format is determined by the customer request format. Supported formats include ISO20022 camt.053.001.02 / camt.052.001.02 (camt.052/camt.053) - responses to the camt.060.001.03 request. Detailed implementation guidelines for ISO20022 account statement request and response XML can be found from ISO 20022 Message Implementation Guide on Swedbank Gateway info page in Internet Bank business.

The reporting request will be rejected if ToDate is missing.

Additional logic:

RequestedMessageNameIdentification= camt.052.001.02 and FromDate=date in the past and ToDate=date in the past: Error „Mismatch of selected message type and report period“
RequestedMessageNameIdentification= camt.052.001.02 and FromDate=date in the past and ToDate=today: Send camt.052.001.02 for current day and camt.053.001.02 for past dates
RequestedMessageNameIdentification= camt.053.001.02 and FromDate=today and ToDate=today: Error „Mismatch of selected message type and report period“
RequestedMessageNameIdentification= camt.053.001.02 and FromDate=date in the past and ToDate=today: Send camt.052.001.02 for current day and camt.053.001.02 for past dates

Periodic statement

It is possible for a customer to get an account statement automatically according to global schedule bank is using for all regular statements (this is once per day and bank will try to send it before midday).

Statement will be generated for previous day from 00:00 until 24:00.
If no transactions are made on that day, empty statement with just balance information will be generated by default. But customer can also specify in the service agreement, that statement without transactions is not generated.
Periodic statement is sent in one or more responses with one response containing up to 10 000 transaction records. This is a current limit which might change. Limit can differ based on the format selected in the SGW agreement.

Other Bank's account statement

This service is provided in accordance with the rules of the PSD2 framework regulation and service availability is depending on another service provider´s API´s accessibility and only request based/REST API.

To see other bank´s accounts via Gateway channel Customer/Operator End Client need provide consent via Swedbank Internet bank for business.

Customer can use this service to request a list of transactions of one specific account on another bank (example domestic SEB) in a given period.

Account statement is returned in one or more responses with one response containing up to 1000 transaction records (pagination apply according to ISO 20022 standard). This is a current limit which might change.

Other bank's periodical statement

It is possible for a customer to get another bank´s account periodic statement automatically according to global schedule bank is using for all Swedbank regular periodical statements sending via gateway channel (this is once per day during the night and bank will try to send it automatically as push messages). Statement will be generated for previous day from 00:00 until 24:00. If no transactions are made on that day, empty statement with just balance information will be generated by default.

Customer can use this service to get a list of transactions of one specific bank and exact account from another bank.

Currency Exchange

Customer can use Currency exchange service to request currency exchange on given account. It is possible to specify worst acceptable rates for both currencies in currency exchange request and SGW will send error if any of rates offered by bank is not good enough to match them.

SGW will also send error in cases when:

there are not enough funds in sell currency on account
transactions are disabled in bank information system
given BIC does not belong to Swedbank.
NB: Currency exchange service similarly to payments service requires that request file is signed at least by one user with Signature certificate. User Signature certificates define users and their access rights to the accounts that currency exchange is made from.

Client must establish daily and monthly limits for each user who is permitted to make transactions (such as payment and currency exchange). Client´s daily and/or monthly limits extent to the all the users and it is mandatory to set if payment signing is in use in client ERP system.

Payment 

Payment is a customer-initiated query. Customer can send one or more payments with one request file. Payment request is sent in one XML file. Swedbank Gateway responds with one or more status reports as payments are being processed.

Customer's information system must be ready to split one payment request file into several smaller request files in case the original request file size exceeds the current maximum incoming message file size set in the channel. Keep in mind that different formats produce different size files per same amount of transactions!

There are 3 possibilities to sign/confirm payment(s):

Accept payments in the Internet Bank for Business Clients in the list of payments.
Payment signing in Client's ERP.
Payment signing via Payment signing API
The second and third possibility requires adding users to SGW agreement.

Based on European Union´s SEPA regulation only ISO20022 format is supported. Supported is ISO20022 format pain.001.001.09 messages with according to pain.002.001.10 response messages.

Currently, we also support pain.001.001.03 and pain.002.001.03 in parallel until November 15th 2026. Therefore, if you continue using pain.001.001.03 and pain.002.001.03 today, please consider that some additional development work will be required in the future for pain.001.001.09 and pain.002.001.10.

Detailed implementation guidelines for supported formats and validation XSDs can be found at ISO20022 Description for Payments on Swedbank Gateway product page:

EE: https://www.swedbank.ee/business/d2d/ebanking/gateway
LV: https://www.swedbank.lv/business/d2d/ebanking/gateway?language=ENG
LT: https://www.swedbank.lt/business/d2d/ebanking/gateway?language=ENG
Limits
SGW agreement enables setting specific limits for payments that every transaction is checked against:

client's daily limit (reset every midnight);
client's monthly limit (reset at midnight on the first of every month);
user´s daily limit (reset every midnight);
user´s monthly limit (reset at midnight on the first of every month)
Client must establish daily and monthly limits for each user who is permitted to make transactions (such as payment and currency exchange). Client´s daily and/or monthly limits extent to the all the users and it is mandatory to set if payment signing is in use in client ERP system.

If the User exceeds the Daily Limit and/or the Monthly Limit, the Bank does not perform the payment orders prepared by the User. If the Client's Daily and/or Monthly Limit are exceeded, the Bank will not perform payment orders submitted hereunder.

Rules of counting the Limits
Limits are defined in local currency and payment amount is checked against them after being converted into local currency using central bank rate.
If the limits specified for the specific Account differ from the limits specified in the Agreement for all accounts of the Client, the limits specified for the specific Account shall prevail.
In case transaction must be confirmed by two Users (double acceptance condition), payment sum is always deducted from the second accepters' limits.
The limits shall not be applied, where:
the funds are transferred between the Client's accounts (belonging to the same company within the Bank);
the Client buys or sells currency (Currency Exchange);
the payment order must be accepted by second Users, in which case the limits shall not be applied to the first signer;
conclusion of agreements, except when the Bank decides to limit certain conclusion of agreements because of security reasons or other important reasons.
Double acceptance condition
Transactions to be accepted by two user/persons
For each user it is possible to set specific double acceptance condition per account/transaction type. Double acceptance condition means that transaction is accepted by Swedbank for processing only if two accepters with appropriate rights have signed the payment file and the limits set for the last signer (i.e. usable daily and monthly limit) are enough to cover the payment. If a certain sum is entered to double acceptance filter starting from Sum, then the second signature is required only for payments equal to/exceeding that sum. Signature container file with payment request XML can include many payments in file and multiple signatures. Further checks will be done:

If multiple users have signed payment file request, then after sending via SGW will check double acceptance condition fulfilment and limits of users in the order of signatures - earliest (time-wise) first.
If first signer has no double acceptance required for unique payment inside signature container for account involved, and all limit checks are successful, then further signatures are ignored.
If first signature has double acceptance required for unique payment inside signature container for account involved, SGW checks next signature - if that one does not have double acceptance required for that account, the payment will be processed.
If both signatures have double acceptance required for the account, they will be checked for limits. If no user has enough available limit left for given payment, that payment will fail with error code showing whether it was daily limit or monthly limit that was not enough (daily limit is checked first).
SGW will repeat the cycle described (points 1-3) per each contained payment through signatures in order of signing until double acceptance condition is fulfilled by 2 signatures OR a signature without a double acceptance requirement with enough available limit is found. Double accept limit is validated on file level, although monitoring single transactions in the file eg payments from that amount on need to have second signer. But as signature can be given only to the file thus to all payments in the file, even if only one payment is bigger, the file will be rejected. General limits as daily and monthly limit are accumulating and deducted from the second signer if the first signer second payment or general payment limit are not sufficient.

Duplicity check
If payment with same data for fields:

Data	Field in ISO 20022 MIG for payment initiation
Beneficiary account	CreditorAccount/Identification/IBAN -> DbtrAcct/Id/IBAN
Remitter account	DebtorAccount/Identification/IBAN -> DbtrAcct/Id/IBAN
Payment amount	InstructedAmount -> InstdAmt
Currency	InstructedAmount -> Ccy
Beneficiary Reference Number	Reference -> Ref
Payment Details	Unstructured -> Ustrd
is received by SGW within 24 hours, then channel marks it automatically as duplicate payment and it will be additionally verified. After that the bank sends final status of payments (accepted ACSC or rejected RJCT). For optional fields such as Payment details and Beneficiary Reference Number „same data „means

a) filled for both and with equal value OR

b) not filled for both.

Swedbank ISO2022 Description for Payments you can find in Swedbank gateway page.

Payment signing in Clients ERP
This service enables send signed payment .asice file from customer ERP to Swedbank. Information about containers and electronic signature you can find from chapter "Authentication and security" and from GitHub

This possibility requires adding agreement limit and users to SGW agreement and your ERP to enable transactions file signing possibility with Signature certificate (example from ID-card, Q Smart-ID, Mobile-ID or others).

Client must establish daily and monthly limits for each user who is permitted to make transactions. Client´s daily and/or monthly limits extent to the all the users and it is mandatory to set if payment signing is in use in client ERP system.

If the User exceeds the Daily Limit and/or the Monthly Limit, the Bank does not perform the payment orders prepared by the User. If the Client's Daily and/or Monthly Limit are exceeded, the Bank will not perform payment orders submitted hereunder.

SGW agreement enables setting specific limits for payments that every transaction is checked against:

client's daily limit (reset every midnight);
client's monthly limit (reset at midnight on the first of every month);
user´s daily limit (reset every midnight);
user´s monthly limit (reset at midnight on the first of every month)
The SGW agreement should be located using AgreementId and the Transport certificate (as it is done with other services) while user's Signature certificates define users and their access rights to the accounts that payments are made from.

Payment signing via Payment Signing API


This service enable customer to sign and execute payment messages on the bank side with simple REST calls via mTLS connection which means that it is not needed to implement payment signing to ERP system. Payment AP is synchronous RESTful API.

Customer can send payment data in ISO20022 xml format. For payment data validation (characters, signature, limits, duplicity check) applies the same rules that are descried under previous chapter (Payment service). Payment API service supports all ISO payment types (Mass, foreign payments, more than 1 payment per file).

PS: Only one file containing max 500 payments, is allowed to be added for each request.

Bank service will create signed ASiC-E via Payment API. Customer is signing payment with qualified signature certificate (only Q Smart-ID and M-ID are supported). After signing of payment container it is possible to execute payment directly via API.

For signing with physical signature tokens (Smartcard/ID-card or USB token/Seal), the container needs to be created in the IS with separate ID software or signature container creation libraries. A possible solution can be found https://www.id.ee/en/rubriik/for-developer/.

Currently there are 6 endpoints for payment API.

Request URL depends on service step. Base url: https://psd2.api.swedbank.com/partner/v1/sgw/

Import payment xml for signing
This endpoint can be used for payment file signing initiation if payment file is not signed on ERP side.

POST [base-url]/signable-payments?client_id=<API Key> : to import ISO20022 payment xml file for signing

HTTP header: X-Request-ID, X-Agreement-ID, Date and Content-Type are mandatory

Request parameter API Keyis mandatory

Request body consist of payment xml data in ISO20022 pain.001.001.03 or pain.001.001.09 format (example under Payment service description).

In response bank provides signablePaymentId and info for next request

File signing
These endpoints allow to initiate file signing. Supported devices are qualified Smart-ID (use smart-id-signatures) or Mobile-ID (use mobile-id-signatures).

If there are multiple signers, then for each signer separate POST request should be performed. In response bank provides a status of signer signature.

POST [base-url]/signable-payments/{signablePaymentId}/smart-id-signatures?client_id=<API Key> : to add Smart-ID signature

HTTP header: X-Request-ID, X-Agreement-ID, Date  and signablePaymentId are mandatory
Request parameter API Keyis mandatory
Request body: signer personal ID (registrationCode) and country code of personal ID (identityCountry).
POST[base-url]/signable-payments/{signablePaymentId}/mobile-id-signatures?client_id=<API Key> : to add Mobile-ID signature

HTTP header: X-Request-ID, X-Agreement-ID, Date and signablePaymentId are mandatory
Request parameter API Keyis mandatory
Request body: signer personal ID (registrationCode), country code of personal ID (identityCountry), phone number (phoneNumber) and customer language (language) .

Transaction notifications

It is possible for a customer to get a notification about every transaction that is booked on a given account. This chargeable service has to be activated for every account through SGW agreement and customer can choose to be notified about incoming payments or outgoing payments.

Additionally, notifications can be filtered by currency and amount – only transactions in certain currency (EUR or all currencies) and exceeding the specified amount in that currency will send a notification.

Detailed implementation guidelines for ISO20022 notification message request and response can be found from ISO 20022 Message Implementation Guide on Swedbank Gateway info page in Internet Bank business and here: XSD validation.

For every transaction matching the filter set in SGW agreement a message is created for each ERP certificate assigned to the agreement. Message is waiting to be picked up for 48 hours, after which it will be deleted from the system.

Request
There is no POST request – bank is generating these messages based on SGW agreement details.

Response
Transaction notifications messages can be found from common inbox: GET https://psd2.api.swedbank.com/partner/v1/sgw/messages?client_id=<API Key>

HTTP header: X-Request-ID and Date are mandatory
Parameter API Keyis mandatory
The response contains:

Body – xml file
Header X-Request-ID – Request ID set by the Bank (alert). .
Header X-Tracking-ID – Bank side message ID.
Header X-Agreement-ID - Swedbank Gateway agreement number provided by the bank
Header Date - Date and time of response
Description of fields
Field name		Description	
X-Request-ID	AN(255)	Message request ID set by the Bank that refers to the service	1..1
X-Agreement-ID	N	Swedbank Gateway agreement number provided by the bank that uniquely identifies the agreement	1..1
X-Tracking-ID	AN	Bank side message ID	1..1
Date	AN	Date and time of request/response in RFC 1123 Full dates format e.g. Tue, 21 April 2023 14:23:49 GMT	1..1
API Key (client_id)	AN	Customer identification number (API Key) generated by Developer Portal during application onboarding that uniquely identifies customer application in SGW API	1..1
All transaction notification messages are in format XML camt.054.001.02.

Transaction notification response differs from statement response only by different container element (Notifications instead of Statement) and missing statement-specific elements (Period, Partial and starting-ending balances for currency).

More info can be found from ISO 20022 Message Implementation Guide or from SGW info page in iBank Business under Useful Information.

E-invoices

Before using this service, it is important to know the terms and conditions of e-invoicing service in the country of operations. Although e-invoice operations exist in all Baltic countries, local implementations differ both from dataset and service range point of view.

The applicability of the current E-invoice format may be verified using the XSD schema available on the website of Bank E-invoice xsd.

PS: This service differs from other SGW services because of request parameters. For all requests it is mandatory to add 'eInvoiceAgreementId' and country.

Estonian e-invoice
Currently supported e-invoice format in Estonia are 1.11 and 1.2. Estonian e-invoice format descriptions can be found at https://www.swedbank.ee/static/pdf/private/d2d/payments/Tehnilised_tingimused(e-arve_saatmine)_ENG.pdf .

Latvian e-invoice
Currently supported e-invoice formats in Latvia are 1.0 and PEPPOL BIS 3.0. Latvian e-invoice format descriptions can be found at: https://www.swedbank.lv/static/pdf/business/cash/einvoices/Latvian_Einvoice_technical_conditions.pdf

Lithuanian e-invoice
Currently supported e-invoice format in Lithuania is 1.1.

Swedbank specific e-invoice technical conditions, including examples, can be found here: https://www.swedbank.lt/static/pdf/business/cash/einvoices/info/Technical-Conditions-of-E-invoice-Swedbank.pdf.

Swedbank Gateway always notifies seller/operator upon receipt of e-invoices with VEA/FEI file.

If file was accepted and no faulty e-invoices found, then empty response is returned
If file was incorrect (missing xsd, unparseable xml) then filebased errorcode is returned in header attribute fileFailReason.
If file included faulty e-invoices (missing internetbank agreement, incorrect account etc), then response file will include Invoice block with errorcode and multiple errorcodes are to be expected.
NB! Full list of e-invoice error codes is described in E-invoice technical descriptions, which can be accessed on E-invoice product page in Swedbank's public website of each country (EE,LV, LT).

Generated link will open pre-filled payment form where payment can be executed from one of Estonian, Latvian or Lithuanian banks.

  "details": "string",

  "language": "string",
  "reference": "string"
}

Operator report

Bank initiated notification type report about conclusion and closure of customer agreement, sent to the operator by default. Bank should be informed in case operator does not want to get the reports.

Report enables operators to receive SGW agreement id for future SGW services usage by the concrete customer. Customer identification can be done by country, registration number and name match.

Request
There is no POST request – bank is generating these messages based on customer agreement conclusion or closure.

Response
Operator report info can be found from common inbox: GET https://psd2.api.swedbank.com/partner/v1/sgw/messages?client_id=<API Key>

HTTP header: X-Request-IDand Date are mandatory
Parameter API Key is mandatory
The response contains:

Body – xml file
Header X-Request-ID – Request ID set by the Bank (CUSTOMER_AGREEMENT_MANAGEMENT).
Header X-Tracking-ID – Bank side message ID.
Header X-Agreement-ID - Swedbank Gateway agreement number provided by the bank
Header Date - Date and time of response
Messages are currently sent one per agreement.

xml. body response examples for new agreement and agreement closure you can see below.

Example of New agreement

<?xml version="1.0" encoding="UTF-8"?>
<GatewayAgreements xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
                   xsi:noNamespaceSchemaLocation="agreementChanges.xsd">
  <Agreement id="123">
    <Country>EE</Country>
    <Name>Legal Name</Name>
    <RegistrationNumber>45789</RegistrationNumber>
    <TimeStamp>2019-01-01T13:10:00</TimeStamp>
  </Agreement>
</GatewayAgreements>

Example of Agreement closure

<?xml version="1.0" encoding="UTF-8"?>
<GatewayAgreements xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
                   xsi:noNamespaceSchemaLocation="agreementChanges.xsd">
  <Agreement id="123" closeAction="BANK">
    <Country>EE</Country>
    <Name>Legal Name</Name>
    <RegistrationNumber>45789</RegistrationNumber>
    <TimeStamp>2019-02-01T16:10:07</TimeStamp>
  </Agreement>
</GatewayAgreements>
NB! Consider future possibility of additional data fields and combining multiple agreements into one XML (duplicating "Agreement" element).

Link to prefilled Gateway Operator agreement conclusion


Externally usable link to Swedbank Internetbank Business for SGW Operator agreement conclusion with prefilled Operator data.

Link will navigate to Swedbank Internetbank login page and after successful login to Swedbank Gateway Operator agreement conclusion screen. Operator is preselected.
Link is operator specific and selection is done based on operatorID provided in link (SGW operator agreement number).
Customer must have representation rights in Company Registry or Swedbank Database, to conclude an agreement.
If Operator agreement is not found (operatorID don’t exist/was written wrongly) or customer al-ready has agreement with this Operator, then following error message is presented: “Pre selecting operator failed! Verify if You already have this operator agreement! Agreement con-cluding is not allowed”.

EE: https://www.swedbank.ee/business/d2d/ebanking/gateway/conclude?operatorId=operator_agreement_id

Markets transaction reports


Markets transaction reports – is a service provided via Swedbank Gateway channel. Technical specifications for supported Transaction types are described in separate document, provided by Swedbank Markets.

Documents are available on Swedbank Gateway product page: EE, LV and LT

This service delivers reports via SGW for supported Transaction types, whenever Client executes a Transaction in accordance with FTMA (Financial Transactions Master Agreement).

To start using the Service, the Client must:

Conclude Swedbank Gateway service agreement;
Request provision of the Markets Transaction Service by completing and signing appropriate section of FTMA;
Perform necessary integration(s) of IT systems to comply with conditions of this Service and SGW Service.
Each Transaction type has a predefined XML structure, corresponding to the type of the financial instrument being traded. The structure and definitions of XML elements for each Transaction type are provided in the separate document, provided by Swedbank Markets.

Currently supported Transaction types are:

Foreign Exchange Spot transaction
Foreign Exchange Forward transaction
Foreign Exchange Swap transaction
Foreign Exchange Option transaction
Deposit transaction
Collateral Deposit transaction
Futures transaction
By using this Service, the Client acknowledges that information and/or Transaction reports provided under this Service do not replace or enhance the information and/or reports required to be provided in accordance with the conditions set by FTMA (and/or its Annexes).

In case of any discrepancies in information being provided, the Client must always rely on information provided in accordance with the clauses of FTMA which govern the provision of such information.
